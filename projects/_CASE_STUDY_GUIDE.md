# Case Study Guide

This document defines the standard for creating project case studies in this portfolio.

The goal is to ensure every project follows a consistent structure, writing style, and presentation.

---

# Philosophy

Every case study should answer one question:

> "Can this person solve real business problems?"

Focus on the business outcome—not just the technologies used.

Recruiters care about:

* Business problem
* Your role
* Your decisions
* Technical implementation
* Results
* Lessons learned

Not simply a list of tools.

---

# Source of Truth

Every project begins as a Markdown document.

```
project-name.md
        ↓
Copy _TEMPLATE.html
        ↓
Generate project-name.html
```

Never edit `_TEMPLATE.html`.

Markdown is always the source of truth.

If a case study needs updates:

1. Update the Markdown.
2. Regenerate the HTML.
3. Verify links and formatting.

---

# Naming Convention

Markdown

```
project-name.md
```

HTML

```
project-name.html
```

Use lowercase letters and hyphens.

Examples

```
vfa-recruitment-funnel.md
algorithmics-lms.md
vimenture-dynamics365.md
```

---

# Required Sections

Every case study should include:

1. Project Overview
2. Business Problem & Context
3. My Role
4. Technologies & Integrations
5. Technical Implementation
6. Solution & Approach
7. Results & Business Impact
8. System Architecture / Workflow Diagram
9. Ongoing Support & Maintenance
10. Lessons Learned
11. Future Improvements

---

# Writing Style

Write professionally but naturally.

Do:

* Explain business decisions.
* Explain why technologies were selected.
* Mention challenges honestly.
* Describe your contribution clearly.
* Highlight measurable outcomes whenever possible.

Avoid:

* Buzzwords without evidence.
* Marketing language.
* Exaggerated claims.
* Copying AI-generated corporate jargon.

---

# Images

Prefer:

* Architecture diagrams
* Workflow diagrams
* Dashboard screenshots
* CRM screenshots (with sensitive information removed)
* Database diagrams

Avoid generic stock images whenever possible.

If screenshots are unavailable, use a simple workflow diagram instead.

---

# Metrics

Whenever possible include measurable results.

Examples:

* Hours saved
* Processes automated
* Users supported
* Records migrated
* Leads imported
* Error reduction
* Time savings
* Data quality improvements

If exact numbers are unavailable, describe qualitative improvements honestly.

---

# Internal Links

Every case study should include navigation back to:

* Home
* Featured Projects
* Contact

Featured Project cards on the homepage should link directly to the corresponding case study.

---

# Copilot Workflow

When generating a new case study:

1. Read the Markdown file.
2. Read `_TEMPLATE.html`.
3. Create a new HTML page.
4. Copy the template structure.
5. Replace placeholder content using the Markdown.
6. Preserve layout, styling, responsiveness, navigation, and scripts.
7. Do not modify `_TEMPLATE.html`.
8. Do not modify existing case studies.
9. Verify all links before finishing.

---

# Quality Checklist

Before committing:

* Markdown updated
* HTML generated
* Navigation works
* Meta title updated
* Meta description updated
* Hero section updated
* Placeholder text removed
* Images verified
* Links tested
* Homepage card updated
* CHANGELOG updated
